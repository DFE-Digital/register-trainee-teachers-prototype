Formatting variations

Here is a page showing a couple of variations of structuring the CSV / HESA guidance content in Register for a discussion within the team.

  • Option 1: Summary cards
  • Option 2: Table
  • Option 3: Patterns used in API ref docs

Option 1: Summary cards

We use them throughout Register, but semantically, they're correct as that's how definition lists are formatted within GOV.UK. We have a large amount of guidance for the CSV fields, which this method scales well to accommodate.

It may be worth including some navigation within the page for users to look up a specific field. This hasn't been looked into yet, but it could be through links or a search within a sidebar or at the top of the page.

Option 2: Table

We currently use tables for HESA guidance; they use vertical space better.

However, as you can see from these examples of new content, they don't scale well for much of it.

I've included a few examples below: a row without any data, one with lots of content that is hard to read when wrapped in a column, and a few shorter examples.

I've also explored horizontal tables, which didn't work but led me to the definition list used within the summary cards above.

Option 3: Patterns used in API ref docs

I have considered using the template and styling we use in the api-reference docs.

The standard headings and paragraphs would work, but we'd have similar issues if we continued incorporating tables (see above). I have mixed feelings about hiding lots of content within a details component.

It would be better to use an index page with links to separate pages if it was deemed necessary from a user's perspective.
